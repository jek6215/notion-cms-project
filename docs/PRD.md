# Notion 공지사항 보드 MVP PRD

## 핵심 정보

**목적**: Notion을 CMS로 활용해 비개발자도 직접 콘텐츠를 관리할 수 있는 팀/스터디/사내 공지 게시판 구축
**사용자**: Notion으로 공지를 작성하는 담당자 + 공지를 열람하는 팀원/구성원

---

## 사용자 여정

```
1. 공지사항 목록 페이지 (메인)
   ↓ 카테고리 탭 선택 (선택적)

2. 필터링된 공지 목록 표시
   - 중요 공지 상단 고정 표시
   - 카테고리 필터 적용됨
   ↓ 공지 카드 클릭

3. 공지사항 상세 페이지
   - 제목, 게시일, 카테고리, 본문 표시
   ↓ 뒤로가기 버튼 or 브라우저 뒤로가기

4. 공지사항 목록 페이지 (복귀)
```

## 기능 명세

### 1. MVP 핵심 기능

| ID | 기능명 | 설명 | MVP 필수 이유 | 관련 페이지 |
|----|--------|------|---------------|------------|
| **F001** | 공지 목록 조회 | Notion DB에서 공개여부=true인 공지를 게시일 내림차순으로 조회 | 서비스 핵심 기능 | 공지사항 목록 페이지 |
| **F002** | 중요 공지 상단 고정 | 중요여부=true인 공지를 일반 공지보다 항상 상단에 표시 | 긴급/중요 공지 가시성 확보 | 공지사항 목록 페이지 |
| **F003** | 카테고리 필터 | 카테고리 탭 선택 시 해당 카테고리 공지만 표시, 전체 탭으로 초기화 | 공지 분류 탐색 | 공지사항 목록 페이지 |
| **F004** | 공지 상세 조회 | 선택한 공지의 제목, 게시일, 카테고리, 본문 내용 표시 | 공지 내용 열람 | 공지사항 상세 페이지 |

| **F010** | Notion API 연동 | Notion Database API로 목록/상세 데이터 fetch, 서버에서만 실행 | 모든 기능의 데이터 소스 | 공지사항 목록 페이지, 공지사항 상세 페이지 |

## 페이지별 상세 기능

### 공지사항 목록 페이지

> **구현 기능:** `F001`, `F002`, `F003`, `F010` | **인증:** 불필요 (공개 페이지)

| 항목 | 내용 |
|------|------|
| **역할** | 서비스 메인 페이지 - 전체 공지를 한눈에 확인하고 필터링 |
| **진입 경로** | 직접 URL 접근 (루트), 헤더 로고 클릭, 상세 페이지에서 뒤로가기 |
| **사용자 행동** | 공지 목록 스캔, 카테고리 탭 클릭으로 필터링, 관심 공지 카드 클릭 |
| **주요 기능** | - 공개여부=true 공지 목록 렌더링 (F001, F010)<br>- 중요 공지 상단 고정 + 시각적 구분 배지 표시 (F002)<br>- 카테고리 탭 UI (전체 포함, Notion DB 카테고리 값으로 동적 생성) (F003)<br>- 탭 선택 시 해당 카테고리 공지만 표시 (F003)<br>- 공지 카드: 제목, 요약, 게시일, 카테고리 배지 표시<br>- **공지 카드 클릭** → 상세 페이지 이동 |
| **다음 이동** | 카드 클릭 → 공지사항 상세 페이지 |

## 데이터 모델

Notion Database를 직접 사용하므로 별도 DB 없음. Notion 필드 구조:

### Notice (Notion Database)

| 필드 | 설명 | Notion 타입 |
|------|------|-------------|
| id | 페이지 고유 ID | Notion Page ID |
| 제목 | 공지 제목 | Title |
| 요약 | 목록 카드용 짧은 설명 | Rich Text |
| 카테고리 | 일반/업데이트/이벤트 등 | Select |
| 중요여부 | 상단 고정 여부 | Checkbox |
| 게시일 | 공지 날짜 | Date |
| 공개여부 | 목록 노출 여부 | Checkbox |

> 본문 내용은 Notion 페이지 블록(Page Content)으로 관리. 상세 조회 시 `blocks.children` API로 별도 fetch.

## 기술 스택

### 프론트엔드 프레임워크

- **Next.js 16** (App Router) - SSR/SSG, Notion API는 서버 컴포넌트에서만 호출
  - `params`, `searchParams` → 반드시 `Promise` 타입, `await` 필수
- **TypeScript 5.6+** - 타입 안전성
- **React 19** - UI 라이브러리

### 스타일링 & UI

- **TailwindCSS v4** - 유틸리티 CSS
- **shadcn/ui** - Badge, Card, Tabs, Skeleton 컴포넌트 활용
- **Lucide React** - 아이콘 (Pin, Calendar, Tag 등)

### CMS & 데이터

- **Notion API** (`@notionhq/client`) - Database query + Page blocks fetch
- 서버 컴포넌트에서만 호출 (API 키 노출 방지)
- 캐싱: route segment config `export const revalidate = 60` (페이지 레벨 ISR)
  - `@notionhq/client`는 자체 HTTP client를 사용하므로 Next.js `fetch()` 캐시와 자동 통합되지 않음

### 블록 렌더링 전략

상세 페이지 본문은 Notion 블록을 직접 렌더링하는 방식으로 구현 (외부 의존성 최소화, React 19 호환성 확보).

**MVP에서 지원할 블록 타입**:

| 블록 타입 | 구현 방식 |
|-----------|-----------|
| paragraph | `<p>` + Rich Text 렌더링 |
| heading_1/2/3 | `<h1>` / `<h2>` / `<h3>` |
| bulleted_list_item | `<ul><li>` |
| numbered_list_item | `<ol><li>` |
| code | `<pre><code>` |
| image | `<img>` (external/file URL) |
| quote | `<blockquote>` |
| callout | 아이콘 + 텍스트 박스 |
| divider | `<hr>` |

**미지원 블록 처리**: "이 콘텐츠는 Notion에서 확인해주세요" fallback 메시지 표시

**Rich Text 처리**: bold, italic, strikethrough, code, link, color 지원

### 에러 핸들링

| 시나리오 | 처리 방식 |
|----------|-----------|
| Notion API 호출 실패 | "공지사항을 불러올 수 없습니다" 메시지 + 새로고침 버튼 |
| 존재하지 않는 공지 ID | `notFound()` → `not-found.tsx` |
| 공개여부=false 공지 직접 접근 | `notFound()` → `not-found.tsx` |
